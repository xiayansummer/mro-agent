# 异步批量外部比价 — 设计文档

> 日期:2026-06-15 · 状态:待评审 · 取代旧「批量询价(库内匹配)」(入口已隐藏,commit cd0fccf)

## 1. 目标与非目标

**目标**:用户上传一份采购清单(Excel/CSV,数十行量级)→ 系统对每一行做**外部平台(京东工业品 / 震坤行)实时比价** → 结果逐行回填,用户可关页面、回头看、跑完收到提示。

**非目标(明确不做)**:
- 不做库内 SKU 匹配垫底(产品已转向外部实时价,库内价不再作为批量数据源)。
- 不追求"秒出";批量本质是异步滴灌(见 §7 吞吐)。
- 不做大批量(上百行)的快速比价 —— 那是未来 D(官方 API)的范畴,本设计架构上为其留口但 v1 不实现。

## 2. 核心约束:风控决定一切

外部比价靠浏览器扩展用用户已登录账号打开平台搜索页采集。**高频访问会触发平台风控**(实测:心跳每分钟开一次京东页 = 60次/小时即被风控)。因此批量**不能**把 N 行一次性灌给扩展猛抓。

> 现状隐患:扩展 `pollAndRunNextTask` 的 `MAX_PER_POLL=20` 会连续猛抓,一旦队列里有 N 条会几分钟内打 20 次京东搜索 → 必触发风控。**本设计的工程核心就是把"猛灌"改成"服务端限速滴灌"。**

## 3. 架构:执行器无关的批量层

```
上传清单 ─→ 解析(复用 inquiry 解析器) ─→ 建 batch + N 个比价任务入队
                                                  │
                                          ┌───────┴────────┐
                                   v1 执行器            未来 D 执行器
                              (扩展限速滴灌)        (官方 API 服务端批量)
                                          └───────┬────────┘
                                          结果落 items_json
                                                  │
                                 批量进度页轮询聚合 ─→ 逐行回填 + 完成提示
```

「上传→解析→batch→N任务→结果回填→进度页」这套**执行器无关**;v1 执行器是扩展滴灌,将来谈下官方 API 后只换执行器,UI/数据模型不重写,67 行可从数小时降到几秒。

## 4. 复用 / 新建

| 直接复用 | 新建 / 改动 |
|---|---|
| `inquiry.py` 的 Excel/CSV 解析(品名/品牌/型号/数量) | `comparison_batches` 表 + `comparison_tasks.batch_id` 列(迁移 SQL) |
| `build_comparison_structure`(已修鲁棒性 8ca12c7) | 批量创建服务:逐行解析→结构→任务 |
| `comparison_tasks/subtasks` 队列 + 落库 + `comparison_ranker` | **限速派发**:`fetchNextTask` 改为按 用户×平台 限速(风控核心) |
| 扩展轮询/采集/回传(**扩展代码不动**) | 批量状态 API + 批量页 UI(上传+进度表) |
| `ComparisonTaskCard`(每行结果渲染) | 完成提示(站内,v1)|

## 5. 数据流(逐步)

1. 上传 → 复用解析器得 N 行(品名/品牌/型号/数量)。
2. 建一条 `comparison_batches`(id, user_id, status, created_at);每行建一个 `comparison_task`(挂 batch_id)各含 jd+zkh 子任务。
3. **逐行解析**:用 `build_comparison_structure` 把"脏品名"抽成结构。**在派发前逐行解析**(随滴灌摊开 LLM 成本,不一次性堆);LLM 抽空时已有兜底(回退原始品名,见 8ca12c7),不丢行。
4. 子任务入队,状态 `queued`。
5. 扩展每分钟轮询 → **服务端限速派发**逐条放行(见 §6)。
6. 采集结果经 `rank_external_offers`(含 disliked 剔除)落 `items_json`。
7. 批量页轮询 batch → 聚合 N 个 task 状态 + 结果,逐行回填。

## 6. 限速派发(风控核心,最高风险单元)

把限速放在**服务端**(不改扩展、可单测、可随时调):

- `fetchNextTask` 改为**平台感知 + 限速感知**:为某 (用户, 平台) 派发下一条子任务前,检查该平台**上次派发时间**;距今 < `MIN_INTERVAL`(起步保守 ~120–180s,带 ±30% 抖动)则该平台**不放行**;京东/震坤行风控独立 → 两平台各自限速、可并行。
  - 上次派发时间**从现有子任务时间戳推导**(该 用户×平台 最近 leased/updated 的子任务),v1 无需新状态表;若不够用再加 `extension_dispatch_state`。
- **退避**:子任务回 `login_required`/验证(风控信号)→ 该 (用户,平台) 进入**冷却**(如 30min)暂停派发,批量页提示"去插件完成验证",冷却后自动恢复。
- 纯函数化:`eligible_platform(last_dispatch_times, cooldowns, now, jitter_seed) -> 平台 | None`,重点单测(正常放行 / 间隔内拦截 / 冷却中拦截 / 双平台独立)。

## 7. 吞吐(诚实的代价)

吞吐 ≈ `3600 / MIN_INTERVAL` 条/小时/平台。保守起步 ~20–30 行/小时:

| MIN_INTERVAL/平台 | 67 行满清单(双平台并行) |
|---|---|
| 150s | ≈ 2.8 小时 |
| 90s | ≈ 1.7 小时 |
| 60s(需实测风控可承受) | ≈ 1.1 小时 |

**一张满清单 = 提交后等 1–3 小时,浏览器需在线**(关了暂停、开了续跑)。速率是服务端常量,真机标定后调;v2 可做"长期无风控就提速、遇风控降速"的自适应。大批量要快仍需 D。

## 8. 交互 / UX

- 入口:重新放出侧边栏「批量比价」(替换隐藏的「批量询价」)。
- 上传页:拖拽 Excel/CSV(复用解析,≤200 行)。
- 进度页:N 行表格,每行状态(排队/进行中/完成/需登录/失败)+ 完成后内联展示该行 top 外部比价结果(可展开 `ComparisonTaskCard`);顶部「X/N 完成」进度。
- **可关页面**:结果落库,回来继续看;跑完(或部分完成)站内提示(侧边栏角标 + 进度页横幅)。真·浏览器推送(web-push)留 v2。

## 9. 错误处理

| 情况 | 行为 |
|---|---|
| 单行无结果 / 平台报错 | 标该行 failed,不阻断其余 |
| 风控 / login_required | 该平台进冷却暂停,提示去插件完成验证,自动恢复 |
| 扩展离线(关浏览器/未绑定) | 派发自然暂停,页面提示"打开浏览器继续",上线续跑 |
| 部分完成 | 已完成行可见,未完成继续滴灌 |

## 10. 测试重点

- **限速器 / 退避状态机**(纯函数):正常放行、间隔内拦截、冷却中拦截、双平台独立、抖动边界 —— 最高风险,重点覆盖。
- 批量创建:N 行 → N 任务/2N 子任务映射;空行/坏行跳过。
- 解析鲁棒性:复用已修的 `build_comparison_structure`(8ca12c7 兜底)。
- 派发集成:队列里 N 条时,单位时间内实际放行数受限(不猛灌)。
